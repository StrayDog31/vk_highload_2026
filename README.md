# Tinder

## 1. Тема и целевая аудитория
### 1.1 Тема
Tinder — международное геосоциальное приложение для знакомств, которое позволяет:

- Находить потенциальных партнеров для общения и романтических отношений на основе взаимного интереса
- Использовать механизм свайпов для быстрой оценки профилей
- Обмениваться сообщениями в чате после образования мэтча
- Использовать рекомендательную систему на основе поведения пользователя и машинного обучения

География и покрытие — всемирно.

### 1.2 Функционал MVP
MVP включает в себя:

- Регистрация и авторизация
- Редактирование анкеты
- Карточка анкеты (отображение фото, имени, возраста и краткой информации о пользователе)
- Механика свайпов
- Рекомендательная система
- Чат
- Уведомление о мэтчах

### 1.3 Целевая аудитория

MAU — 75 млн. человек
DAU — 42 млн. человек

Основные регионы:
- США - 35%
- Латинская Америка (Бразилия, Аргентина) - 10%
- Европа - 49%

Средний возраст: 18–35 лет (ядро аудитории — миллениалы и поколение Z)

Гендерное распределение:
- Женщины: 25%
- Мужчины: 75%

### 3.4 Схемы глобальной балансировки
- Latency-based DNS для API и Gateway трафика: DNS-резолвер пользователя направляется на ближайшую точку присутствия на основе геолокации и сетевой задержки. Используется короткий TTL (60 сек), что позволяет быстро перенаправлять пользователей на другие регионы при сбоях или перегрузке ДЦ.
Применение: Домены api.tinder.com, gateway.tinder.com, static.tinder.com

- BGP Anycast + CDN для медиа-трафика: Ключевые домены медиа-инфраструктуры анонсируются из множества точек присутствия CDN-провайдера с использованием Anycast. Пользователь перенаправляется в ближайший узел сети с минимальным числом хопов. Для доставки фото используется распределённая сеть CDN-узлов, что снижает задержки и нагрузку на центральные ДЦ.
Применение: Домен media.tinder.com

### 3.5 Механизмы регулировки трафика между ДЦ
- Active Health Checks: Постоянно выполняются проверки доступности edge-узлов CDN, API-шлюзов и origin-серверов во всех 6 ДЦ. Проверки включают health check эндпойнты (/health) на API-серверах, проверку задержек ответа БД и мониторинг загрузки CPU/сети. При обнаружении деградации или отказа узел автоматически исключается из DNS-выдачи.

- Least Connections: Длительные WS соединения чатов направляются на ДЦ с меньшим количеством подключений.

- Weighted Round-Robin: Использование весовых коэффициентов в глобальном балансировщике нагрузки для управления долями входящего трафика между ДЦ внутри одного региона:

## 4 Локальная балансировка нагрузки
### 4.1 Схема балансировки нагрузки
Внутри каждого ДЦ маршрутизация запросов разделена на обработку внешнего и внутреннего трафика.

Внешняя балансировка:

L4 (Транспортный уровень):
Используется SLB LVS.
- Задача: распределение TCP-трафика
- Режим: Direct Routing (DR). Обратный трафик от серверов идет напрямую к клиенту, минуя балансировщик. Это важно для тяжелых изображений и предотвращает перегрузку сети на LVS.
- Резервирование: Active-Active (2 + 1 инстанса), health check каждые 5 секунд
- Количество инстансов: 3 сервера на ДЦ (2 активных + 1 пассивных)

L7 (Прикладной уровень)
Используются серверы NGINX.
- Задача: осуществляют TCP multiplexing, поддерживают WS соединения, выполняют SSL Termination (расшифровку HTTPS).
- Резервирование: Active-Active (2 + 1 инстанса),
- Количество инстансов: 3 сервера NGINX на ДЦ

Внутренняя балансировка:
Для внутреннего трафика ипользуется Envoy Sidecar. Каждый микросервис имеет свой локальный Envoy-прокси, который обеспечивает всю логику взаимодействия с другими сервисами.

- Балансировка между инстансами микросервисов
- Retry при временных ошибках
- Таймауты на запросы
- Сбор метрик (Prometheus)
- Балансировка gRPC-трафика

### 4.2 Рассчет количества балансировщиков
Расчет производится для одного ДЦ на основе пиковых показателей:

| Параметр | 	Значение для US East/EU West |
|-------------|----------------------------|
| Суммарный трафик на 1 ДЦ | 46,7 Гбит/с | 
| Трафик L7 (HTTP + WS, без медиа) | 0,04 Гбит/с |
| Пиковый RPS L7 (HTTP + WS) | 42088 RPS |

Расчет балансировщиков L7 (NGINX)
Для L7 учитываются два ограничителя: пропускная способность и SSL Termination.

По пропускной способности:
- Трафик L7 (текст, уведомления, рекомендации): ~0,04 Гбит/с
- Пропускная способность одного NGINX-сервера: 100 Гбит/с
- 0,04 Гбит/с ÷ 100 Гбит/с = 0,0004 → 1 сервер

По SSL Termination:
- Интенсивность новых TLS-соединений принята равной 30% от общего RPS (с учетом постоянных WebSocket-соединений)
- Пиковый RPS: 42088 RPS × 0,3 = 12626 CPS (connections per second)
- При производительности одного сервера 6 676CPS (из примера):

12626 CPS ÷ 6676 CPS = 1,89 → 2 сервера

Итог для L7:
Выбираем «худший» случай — 2 сервера (по SSL Termination).
С учетом резервирования (N + 1):
2 + 1 = 3 сервера NGINX на один ДЦ

Расчет балансировщиков L4 (LVS)
Ограничителем для L4 выступает только пропускная способность сетевых интерфейсов. Для L4 серверов используются сетевые карты 40GbE.

По пропускной способности: Суммарный трафик на ДЦ: 46,7 Гбит/с
46,7 Гбит/с / 40 Гбит/с = 1,17 → 2 сервера

Итог для L4:
Требуется 2 активных сервера. Применяем формулу резервирования N + 1 (Active-Active):
2 + 1 = 3 сервера LVS на один ДЦ (2 активных + 1 пассивных в горячем резерве)

### 6.6 Балансировка запросов

| СУБД | Схема | Обоснование |
|-------------|----------------------------|-------------------------|

| Scylla | Token-aware | Запрос отправляется напрямую на нужный шард (по хешу ключа) |
| FDB | Proxy + Cluster controller (https://habr.com/ru/articles/441270/) | Когда клиент хочет подключиться к БД, он обращается сразу ко всем Coordinators за адресом текущего Cluster Controller. Если большинство ответов совпало, он получает из Cluster Controller полную текущую конфигурацию кластера |

## 7 Алгоритмы

### Рекомендательная система:
Алгоритм строится на двух источниках сигнала с разными весами для наилучшего подбора:

Источник 1: Семантическая близость анкет - 70%:

Каждое текстовое описание анкеты при создании или обновлении преобразуется в эмбеддинг с помощью предобученной модели, которая учитывает контекст и синонимы. Для MVP можено выделять просто ключевые слова. Вектор нормализуется, вычисляется угол между векторами анкет. Для пользователя собирается определенный профиль, основываясь на анкетах за последние N дней. Для каждой анкеты вычисляет угол между ее эмбеддингом и предпочтением юзера. Для нового аккаунта используется эмбеддинг собственной анкеты пользователя.

Источник 2: Поведенческие метрики - 30%:

Соберем профиль активности юзера по этим метрикам:
- длина описания (заполненность анкеты)
- скорость ответа на сообщения
- дневная активность
- количество фотографий
- количество проставленных лайков к свайпам

Все они нормализуются относительно распределения по всей платформе (robust scaling) в диапазон от 0 до 1. Считаем среднее арифметическое этих метрик. Таким образом, пользователи получают кандидатов с похожими поведенческими паттернами.

Источник 3: Ограничения:
Анкеты дополнильно фильтруются по:
- Возрасту
- Региону
- Полу

Далее с учетом этих трех параметров создаем итоговый score = 0.7*S1 + 0.3*S2. Если анкета не проходит источник 3 - выкидываем из подбора.

## 8 Технологии

| Название | Назначение | Обоснование |
|-------------|----------------------------|-------------------------|
| Next JS, Redux, TS | Фронтенд приложения | TS является стандартом в текущих реалиях. Nest - один из ведущих фронтенд SPA фреймворков, Redux удобен для управления состояниями |
| Go | Основные микросервисы бэк | Высокопроизводителен и заточен под асинхронное программирование. Имеет удобные инструменты для интеграции с прочими технологиями |
| Go/Python | Рекомендательная система | У Python ощутимое приемущество в работе с моделями, нежели у Go. Но на Go это тоже возможно |
| PostgreSQL | Основное хранилище | Одна из самых оптимальных реляционных СУБД. |
| ScyllaDB | Хранение сообщений | Эффективная ключ-значение БД. |
| Redis | Хранение сессий | Лучшее резистентное хранилище. |
| FoundationDB | Работа со свайпами и мэтчами | Так же ключ-значение с поддержкой транзакций. Авторешардинг. |
| S3 | Хранение тяжеловесный фото юзеров | Позволяет эффективно хранить тяжелые вещи вне БД. |
| Kafka | Доставка уведомлений | Обеспечивает очередность без потерь и риал-тайм события. |
| Docker | Контейнеризация | Единое окружение сервисов. Упрощает сборку, деплой и внесение риал-тайм изменений в продукт. |
| Kubernetes | Управление контейнерами | Автоматизация деплоя и масштабирования сервисов. |
| Grafana + Victoria Metrics | Метрики | Замена устаревшего Prometheus, куда лучшее масштабирование |
| Nginx | Веб-сервис, L7, SSL | Автоматическая балансировка бэка. Отраслевой стандарт, лучшего решения на рынке не существует. |

Для хранения эмбеддингов можно задействовать векторую БД, но с натяжкой для MVP подойдет и PostgreSQL.
