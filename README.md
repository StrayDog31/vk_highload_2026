# Tinder

## 1. Тема и целевая аудитория
### 1.1 Тема
Tinder — международное геосоциальное приложение для знакомств, которое позволяет:

- Находить потенциальных партнеров для общения и романтических отношений на основе взаимного интереса
- Использовать механизм свайпов для быстрой оценки профилей
- Обмениваться сообщениями в чате после образования мэтча
- Использовать рекомендательную систему на основе поведения пользователя и машинного обучения

География и покрытие — всемирно.

### 1.2 Функционал MVP
MVP включает в себя:

- Регистрация и авторизация
- Редактирование анкеты
- Карточка анкеты (отображение фото, имени, возраста и краткой информации о пользователе)
- Механика свайпов
- Рекомендательная система
- Чат
- Уведомление о мэтчах

### 1.3 Целевая аудитория

MAU — 75 млн. человек
DAU — 42 млн. человек

Основные регионы:
- США - 35%
- Латинская Америка (Бразилия, Аргентина) - 10%
- Европа - 49%

Средний возраст: 18–35 лет (ядро аудитории — миллениалы и поколение Z)

Гендерное распределение:
- Женщины: 25%
- Мужчины: 75%

### 2.2 Технические метрики

Расчёт объёма хранения
Расчёт производится для хранения данных, генерируемых за 1 год использования сервиса.

| Тип файлов | Рассчет на пользователя | Общий объем |
|------------|--------------------------|-------------|
| Текстовое сообщение | 10 сообщений * 26 млн. мэтчей / 42 млн. пользователей * 365 дней * 200 байт = 0,45 МБ |18,9 ТБ |
| Фото (оригиналы) | 5 ед. * 200 Кбайт = 1 МБ | 42 ТБ |
| Превью фото | 5 ед. * 200 Кбайт * 1,5 = 1,5 МБ | 63 ТБ |
| Мэтчи | 26 млн. мэтчей / 42 млн. пользователей * 365 дней * 50 байт = 0,011 МБ | 0,46 ТБ |
| Лайки | 1,6 млрд свайпов * 38% * 50 байт * 365 дней / 42 млн пользователей = 0,27 МБ | 11,3 ТБ |
| Итого | 3,23 МБ | 135,7 ТБ |

Пояснение: Мужчины (75% пользователей) лайкают 46% профилей, женщины (25% пользоателей) в районе 14%. Итого 38% лайков от 1.6 млрд. свайпов.

Сетевой трафик
При расчёте используется коэффициент суточной неравномерности k = 2 для определения пиковой нагрузки.

| Тип трафика | Суточный объём (Тбайт/сут) | Средний трафик (Гбит/с) | Пиковый трафик (k=2) (Гбит/с) |
|-------------|----------------------------|-------------------------|-------------------------------|
| Текстовые данные | 26 млн мэтчей * 10 сообщ. * 200 байт = 0,052 | 0,005 | 0,01 |
| Фото (просмотр анкет) | 42 млн * 100 свайпов * 200 Кбайт = 840 | 77,8 | 155,6 |
| Фото (загрузка) | 5 новых фото * 42 млн * 200 Кбайт / 365 = 0,115 | 0,01 | 0,02 |
| Итого | 840,2 | 77,8 | 155,6 |

Расчёт RPS (Requests Per Second)
RPS рассчитывается по формуле: RPS = (DAU × Действия в сутки) / 86 400.

| Тип запроса | Общее кол-во запросов в сутки | Средний RPS | Пиковый RPS (k=2) |
|-------------|-------------------------------|-------------|-------------------|
| Свайпы (лайки/дизлайки) | 1,6 млрд | 18 519 | 37 038 |
| Отправка сообщений | 26 млн мэтчей * 10 сообщ. = 260 млн | 3 009 | 6 018 |
| Загрузка фото | 42 млн * 0,014 загрузок в день = 588 тыс. | 7 | 14 |
| Просмотр фото | 42 млн * 100 свайпов * 1 фото = 4,2 млрд | 48 611 | 97 222 |
| Итого | 6,06 млрд | 70 146 | 140 292 |

*0,014 загрузок в день = 5 новых фото в год / 365 дней

## 3 Глобальная балансировка нагрузки
### 3.1 Функциональное разбиение по доменам
| Доменное имя | Назначение | Требования к задержке |
|-------------|----------------------------|-------------------------|
| tinder.com/api | Авторизация, настройки профиля, список мэтчей, свайпы | Очень низкая. Пользователь не должен ждать при свайпе или отправке сообщения. |
| static.tinder.com | Js-бандлы, CSS, HTML  | Средняя. Загрузка при первом входе или обновлении приложения. |
| media.tinder.com | Загрузка и получение тяжелых медиа (фото профиля) | Низкая (<200-300 мс) для загрузки превью при свайпе. Загрузка фото может быть чуть медленнее. |
| gateway.tinder.com | Поддержание WS соединений, обмен сообщениями, статусы и т.д. | Очень низкая. Пользователь не должен долго ждать сообщений. |

### 3.2 Расположение ДЦ
Учитывая глобальную аудиторию (США, Европа, Латинская Америка) и требования к низкой задержке,:ДЦ должны быть расположены в регионах с наибольшей плотностью пользователей.

Основные кластеры:
- US East (Северная Вирджиния, Огайо): Обслуживает Северную и, частично, Южную Америку.
- EU West (Франкфурт, Ирландия, Лондон): Обслуживает Западную Европу.
- EU East (Варшава): Обслуживает Восточную Европу.
- US West (Орегон) или Sao Paulo (Бразилия): Для резервирования US East и лучшего покрытия Латинской Америки (если задержки из US East в Бразилию велики, лучше иметь ДЦ в Сан-Паулу).
- APAC (Сингапур): Обсулживает все, что осталось.

### 3.4 Схемы глобальной балансировки
Latency-based DNS для API и Gateway трафика
DNS-резолвер пользователя направляется на ближайшую точку присутствия на основе геолокации и сетевой задержки. Используется короткий TTL (60 сек), что позволяет быстро перенаправлять пользователей на другие регионы при сбоях или перегрузке дата-центров.
Применение: Домены api.tinder.com, gateway.tinder.com, static.tinder.com

BGP Anycast + CDN для медиа-трафика
Ключевые домены медиа-инфраструктуры анонсируются из множества точек присутствия CDN-провайдера с использованием Anycast. Это позволяет маршрутизаторам Интернета автоматически направлять пользователя в ближайший узел сети с минимальным числом хопов. Для доставки фото используется распределённая сеть CDN-узлов, что снижает задержки и нагрузку на центральные дата-центры.
Применение: Домен media.tinder.com

### 3.5 Механизмы регулировки трафика между ДЦ
Active Health Checks
Постоянно выполняются проверки доступности edge-узлов CDN, API-шлюзов и origin-серверов во всех 6 ДЦ. Проверки включают health check эндпойнты (/health) на API-серверах, проверку задержек ответа БД и мониторинг загрузки CPU/сети. При обнаружении деградации или отказа узел автоматически исключается из DNS-выдачи.
